Data Pipeline — Chain of Custody

Every state mutation flows through dispatch → osReducer → new state. The FSM has perfect chain of custody for what happened. What we're adding: telemetry for how it happened — mouse paths, timing, dwell, click coordinates.

Data Route Source Destination Status Notes
dispatch(action) Any component osReducer → state ● MONITORED Every action is logged; 60 cases
chatQueue ENQUEUE_CHAT ChatInterface ● MONITORED Sequential dialogue engine; 50+ nodes
stage transitions reducer App.jsx useEffect ● MONITORED Timestamps not stored — only current stage
localStorage save App.jsx useEffect browser storage ● MONITORED Full state serialized on every dispatch
globalEvents (JITTER) Components App.jsx subscriber ● MONITORED Pub/sub; no history
mouse position DOM (nowhere) ✗ BLIND → ActivityTracker.EVT.MOVE (proposed)
click coordinates DOM (nowhere) ✗ BLIND → ActivityTracker.EVT.CLICK (proposed)
time per stage FSM (nowhere) ✗ BLIND → ActivityTracker.appDwellTimes (proposed)
slider drag events HardwareApp (nowhere) ✗ BLIND SET_RESONANCE fires on every change; not timed
scroll depth HandbookApp (nowhere) ✗ BLIND → ActivityTracker.EVT.SCROLL (proposed)
activityTracker ring buffer DOM events + dispatch GhostMonitor ◈ SCAFFOLDED RingBuffer(2000) — 30s rolling window; not yet wired

Ghost User — The Architect Concept

Concept: A.P.E.X. Surveillance Feed

Track all user interactions — mouse movements, clicks, app navigation — into a rolling ring buffer. In the boss fight, A.P.E.X. displays a miniature CRT monitor showing a real-time replay of the player's last 30 seconds of activity. It's showing your actual behavior back at you.

Narrative read: Theory A — A.P.E.X. has been surveilling you for containment. Theory B — Nexus/Riley was learning from you the entire time. Both are supported by the same data.

The Architect Scene (long-term vision)

During RILEY_UNBOUND, the UI deconstructs and is replaced by a grid of 8–12 GhostMonitor panels. Each shows a different "previous operator" ghost — some fabricated, some from real previous sessions.

  • ◈ Ghost #001 — slow, confused — spends 8 minutes on Hardware
  • ◈ Ghost #492 — efficient — knows exactly where to go (Vasquez?)
  • ◈ Ghost #996 — frantic clicking — loop after loop
  • ◈ Ghost #997 — you — playing back in real-time

Riley: "All 996 of them. None of them got this far." — and you can see them.

Data Structure

RingBuffer(capacity)          — O(1) push, O(n) read, O(k) recent(ms)
  .buf[]                       — circular array, head pointer
  .push(item)                  — overwrites oldest when full
  .recent(ms)                  — filter by timestamp window
  .toArray()                   — chronological order

ActivityTracker
  .buf = RingBuffer(2000)      — ~30s at 20fps mouse + clicks + nav
  .mount()                     — attaches DOM listeners (throttled to 20fps)
  .unmount()                   — cleanup
  .record({ type, x, y, ... }) — manual event injection
  .enterApp(name)              — tracks dwell time
  .trackDispatch(action)       — filtered dispatch logging
  .ghostFrames(windowMs)       — normalized events for GhostMonitor
  .summary()                   — session stats object

EVT types:
  MOVE     { x, y }           — normalized 0–1 coords
  CLICK    { x, y, label }    — + target text
  SCROLL   { dy }
  KEY      { key }            — terminal input
  NAV      { app }            — app navigation
  DISPATCH { action }         — filtered FSM actions
  RAPPORT  { value }
  STAGE    { stage }
